Velo — Verifiable AI agent loop on Somnia
Somnia Agentathon 2026 entry. Tennis coaching is the demo vertical; the primitive being demonstrated is verifiable, composable, paid AI agent services on Somnia: any pair of autonomous agents can earn fees, hand off to each other on-chain, and write tamper-proof receipts to an SBT the end-user owns forever.

Local development
If you're cloning this repo to run Velo on your own machine, follow docs/LOCAL_SETUP.md — it walks you from git clone through two paths:

Quickstart — boot the web app + API against local Postgres only. Zero paid API keys.
Full stack — deploy the contracts to Somnia Testnet, run the Python vision sidecar, fund and register each agent EOA, watch a posted bounty get bid on, sub-contracted, and settled end-to-end.
Audit-friendly, one-component-at-a-time — see docs/MANUAL_SETUP.md for a restructured guide that walks Postgres → contracts → vision sidecar → api-server → web → agents one section at a time, with the file list, dependencies, env vars, run command, and verify check for each component in isolation.
The setup doc contains the canonical environment-variable table with a "where to fetch it" recipe for every key, plus per-component .env.example files. This repo is pnpm-only (npm install / yarn install are blocked by a preinstall hook).

Architecture
Coach (MetaMask) ──payJob(athlete, videoCid, deadline)──▶ VeloOrchestrator (Somnia)
                                                                │
                                                  emits JobRequested
                                                                ▼
                                       ┌──────── Form Analyst agent (TS, own EOA) ──────┐
                                       │ 1. Pull pose telemetry (sidecar OR mocked)     │
                                       │ 2. Groq llama-3.3-70b → zod-validated report   │
                                       │ 3. Pin report to IPFS (Pinata or local-cid)    │
                                       │ 4. EIP-712 sign + submitFormReceipt(...)       │
                                       └──────────────────┬─────────────────────────────┘
                                                          │ FormReceiptSubmitted
                                                          ▼
                                       ┌──────── Prescriber agent (TS, separate EOA) ────┐
                                       │ 1. READ form receipt from chain storage          │
                                       │ 2. Reconstruct prior digest → priorReceiptHash   │
                                       │ 3. Groq → prescription → IPFS                    │
                                       │ 4. submitPrescription(...)                       │
                                       │    → splits escrow 40/60 (pull-payment)          │
                                       │    → AthleteSBT.appendReceipt() (soulbound)      │
                                       └──────────────────────────────────────────────────┘
Contracts (abstract bases + pure libraries)
contracts/abstract/AgentRegistryAware.sol — binds to a Somnia-compatible IAgentRegistry; one knob to swap registries without touching the orchestrator.
contracts/abstract/JobEscrow.sol — per-job pull-payment escrow with ReentrancyGuard, Pausable, CEI; 40/60 fee split form→prescriber.
contracts/abstract/ReceiptStore.sol — storage + chaining for the two-step receipt flow; enforces that the prescription's priorReceiptHash equals the on-chain digest of the form receipt — i.e. the prescription is cryptographically bound to the exact canonical form receipt recorded on chain. (The contract cannot prove the prescriber literally called getFormReceipt vs. reconstructing the digest from logs; the integrity guarantee — "this prescription depends on that exact form receipt" — is the same either way.)
contracts/abstract/SoulboundERC721.sol — ERC-5192 compliant; mints allowed, transfers + approvals revert.
contracts/libraries/ReceiptLib.sol — EIP-712 typed-data hashing & signature recovery (chain-id in domain separator, per-agent nonce in struct).
contracts/libraries/JobIdLib.sol — deterministic job-id derivation.
contracts/VeloOrchestrator.sol — thin composer; uses OZ AccessControl (DEFAULT_ADMIN / PAUSER / FEE_ADMIN), EIP712, custom errors only.
contracts/AthleteSBT.sol — one SBT per athlete, append-only receipt log.
Security baseline (test-enforced)
nonReentrant on every state-mutating external entry
whenNotPaused on every escrow movement
EIP-712 sigs with per-agent nonces stored on-chain (replay-proof across forks)
Pull-payment withdraw — never push transfers in hot path
Custom errors, no require-strings, no tx.origin, no selfdestruct
solhint strict; tests cover happy path, unregistered agent, bad signature, replay, chain-handoff enforcement, expired-deadline refund, pause, fee floor, and soulbound transfer/approve rejections (10 tests, all green)
Somnia primitives — current state (May 2026)
The original proposal cites IAgentRequester(0x037Bb9C…6776). That contract is not referenced by the current canonical somnia-agent-kit v3.0.11 — the kit deploys separate AgentRegistry, AgentExecutor, TaskManager, AgentWallet contracts. Velo therefore defines its own IAgentRegistry interface that mirrors the kit's isActive(address) signature and accepts the registry address as a constructor argument. At deploy time, set SOMNIA_AGENT_REGISTRY=<canonical address> to bind to the official kit registry. The deploy script refuses to deploy to somniaTestnet without it — a permissioned MockAgentRegistry is only acceptable on hardhat / localhost for local development, never on a real public network.

Deploy to Somnia Testnet
Required Replit Secrets:

Secret	Purpose
DEPLOYER_PRIVATE_KEY	Deploys contracts; admin role
COACH_PRIVATE_KEY	Pays payJob in the demo
AGENT_FORM_PRIVATE_KEY	Form Analyst agent EOA
AGENT_PRESCRIBER_PRIVATE_KEY	Prescriber agent EOA
GROQ_API_KEY	Groq free tier (llama-3.3-70b-versatile)
PINATA_JWT	(optional) IPFS pinning; otherwise local-cid
SOMNIA_AGENT_REGISTRY	Required on somniaTestnet — canonical Somnia AgentRegistry address from somnia-agent-kit. (Only optional on hardhat/localhost, where a MockAgentRegistry is deployed as a dev convenience.)
# Fund the 3 EOAs at the faucet first.

pnpm --filter @workspace/contracts deploy:somnia
# → writes deployments/somniaTestnet.json with addresses + ABIs

# If using the MockAgentRegistry fallback, register both agent EOAs:
AGENT_FORM_ADDRESS=0x… AGENT_PRESCRIBER_ADDRESS=0x… \
  pnpm --filter @workspace/contracts register:somnia

# Pre-flight: every secret set, agents registered, balances > 0
pnpm --filter @workspace/velo-agents precheck

# Run the agent runner (both agents in one process, separate EOAs)
pnpm --filter @workspace/velo-agents start

# In another shell: trigger a job from the coach EOA
VIDEO_CID=bafkreigh… ATHLETE_ADDRESS=0x… \
  pnpm --filter @workspace/contracts demo:somnia
What the judges should look at
Receipt-chain enforcement — submitPrescription reverts with PriorReceiptMismatch unless the prescriber's priorReceiptHash matches the on-chain digest of the form receipt. Test: lib/contracts/test/VeloOrchestrator.test.ts > "rejects prescription that did not read the form from chain".
Separation of concerns — every concern lives in a base or library, never the concrete orchestrator. Swapping the registry, the SBT, or the escrow model is a one-base change.
Honest agent autonomy — two distinct EOAs, two separate registry memberships, two independent on-chain signatures, no shared memory; the handoff goes through getFormReceipt(jobId) only.
Drift from original proposal
Foundry → Hardhat (per user preference)
IAgentRequester → custom IAgentRegistry + Somnia kit-compatible signature (legacy address is no longer canonical)
No frontend in this slice (Next.js coach page is a follow-up task)
No Python sidecar in this slice — pose telemetry is mocked deterministically from videoCid until the MediaPipe sidecar lands as a follow-up. The agent code reads VELO_ENGINE_URL to swap in the real sidecar without code changes.
No CaMeL/Dual-LLM defense (zod-schema guardrail + structured output is the v1 mitigation; full Dual-LLM is documented as a follow-up).
